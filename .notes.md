## Login continuation

- We can set the user token to the cookie and set it to httpOnly such that javascript on a user's browser is not going to be able to access the token.
- Setting the user cookie is happens in generateToken so that you can tie it to the res given back

```javascript
//generateToken function that takes userId and the res from the function inputs.
export const generateToken = (userId, res) => {
  //Generate token process using the UUID assigned from the register
  const payload = { id: userId };
  const tokenGenerated = jwt.sign(payload, process.env.JWT_SECRET, {
    expiresIn: process.env.JWT_EXPIRES_IN || "1d",
  });
  //Handle response and assign it a cookie, give it a name and the key generated.
  res.cookie("jwt", tokenGenerated, {
    httpOnly: true,
    secure: process.env.NODE_ENV === "production",
    sameSite: "Strict",
    maxAge: 24 * 60 * 60 * 1000, // 1 day in milliseconds
  });
  return tokenGenerated;
};
```

## Middlewares

What is a middleware?
this is a function that sits between a request and a response.
This ensure authentication and authorisation checks of the request before we get to the endpoint.

We are going to add a middleware to check the user permissions using the jwt and ensure they can perform the operation.

src/middleware/authMiddleware.js --> You can have middleware to a tone of things.

Once we define our function in the middleware, we import it in out routes (watchlistRoutes.js)
router.use(authMiddleware) - This will ensure before any route defined runs, middleware will first.

In the middleware, we do some checks and then tell the initial request to continue and give back a response.

The token is sent through the headers and the check is done.

Standard:
Authentication and authorisation tokens should always be in the header.
Body is just for info to be parsed into the req.

Auth token standard
Authorization Bearer <token>

In a middleware function, it takes req, res and next
next() - this is a function that let's the middleware close and the route to be completed.

Because the userId is being used to generate the token, we can get it in the token instead of passing it in the req.body.

We can grab userId from authentication because it is being used to come up with the token.

Using middlewares we can add things to req part of the request as it goes to the path, and that is why we pass authentication and authorisation through it.

**Key Differences - Zod vs Express**
Type Safety: Zod offers excellent automatic type inference for TypeScript projects, eliminating the need to define interfaces twice. Express-Validator has good type definitions but requires more manual setup.
Sanitization: Express-Validator provides extensive sanitization (trimming, escaping, conversion) built-in. Zod offers limited sanitization, focusing primarily on validation and basic transformations.
Integration: Express-Validator integrates directly into Express route chains (e.g., body('email').isEmail()), making it easy for simple Express apps. Zod is a standalone library that requires custom middleware or wrappers to integrate with Express.
Error Handling: Express-Validator attaches errors to the Express request object (req.validationResult()). Zod provides structured, user-friendly error objects that you handle explicitly via try/catch or safeParse.
Learning Curve: Express-Validator has a low learning curve for those already familiar with Express. Zod has a low-to-moderate curve, offering a clean API but requiring understanding of schema definition and TypeScript inference.
When to Choose Which
Choose Express-Validator for quick integration in plain JavaScript Express apps, when you need extensive input sanitization (e.g., trimming, HTML escaping), or for simple field-level validation without TypeScript.
Choose Zod for TypeScript projects where you want to eliminate type duplication, prefer a modern, zero-dependency library, or need complex nested object validation with strong type safety.
